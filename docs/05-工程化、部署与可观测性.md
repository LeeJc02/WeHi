# 工程化、部署与可观测性

项目真正像“工程项目”而不是“功能集合”，靠的是这部分能力。

## 启动与部署

文件：

- `deploy/compose/docker-compose.yml`
- `Dockerfile.service`
- `Dockerfile.frontend`
- `Makefile`

默认启动方式：

```bash
make start
make refresh
make stop
make smoke
```

本地开发环境会启动：

- `mysql`
- `redis`
- `jaeger`
- `migrate`
- `auth-service`
- `api-service`
- `realtime-service`
- `frontend`

## 为什么这套启动链路值得写进文档

因为它体现的不只是“能跑”，而是：

- migration 在服务之前执行
- 每个服务都有 health check
- 前端依赖后端 ready 之后再启动
- 环境变量通过部署配置集中注入

这说明你考虑了：

- 启动顺序
- 服务依赖
- 环境隔离
- 复现成本

## migration 而不是 AutoMigrate

目录：

- `migrations/`

这点很适合面试时主动强调。

优点：

- schema 演进可审计
- 线上变更风险可控
- 索引和约束改动更明确

比如消息幂等约束的调整就明确落在：

- `migrations/006_message_lifecycle_idempotency.sql`

## 可观测性

文件：

- `internal/platform/observability/otel.go`
- `internal/platform/httpx/httpx.go`
- `internal/realtime/hub.go`

当前已经覆盖：

- HTTP tracing
- GORM / MySQL tracing
- Redis tracing
- WebSocket 连接指标
- 结构化日志
- Prometheus `/metrics`
- Jaeger trace 查看

`Hub` 里还有一个直接可展示的指标：

```go
var wsConnections = promauto.NewGauge(prometheus.GaugeOpts{
	Name: "chat_ws_connections",
	Help: "Current number of live websocket connections.",
})
```

这个细节很好，因为它说明不是只会写业务逻辑，也会把运行态指标暴露出来。

## 后台诊断能力

项目不只是“有 trace”，还有后台诊断接口：

- 消息旅程
- 会话一致性
- AI 审计
- 搜索重建

对应路由在：

- `internal/routes/api.go`

你可以把这部分总结成：

“我不仅实现了链路，还补了面向排障和运维的可视化入口。”

## Docker 与 Kubernetes 部署

项目通过 Docker 完成服务镜像封装，并基于 Kubernetes 承接多服务部署、服务注册发现、健康检查和弹性扩缩。

这部分在架构层面解决的是：

- 多服务独立交付
- 服务实例解耦与弹性扩容
- 配置、密钥和环境隔离
- 发布过程中的可维护性与可回滚能力

## 当前工程化仍然不足的地方

这部分建议你在面试里主动承认。

- `sync_events` 缺少归档与清理策略
- 大群场景下的会话统计和未读聚合还有进一步优化空间
- 网关路由、resume 恢复信息和补偿窗口治理还可以继续细化

主动说这些，反而更像资深工程师。

## 面试可展开的过程问题

### 你是怎么保证多服务项目本地还能稳定启动的

回答重点：

- 统一入口命令
- Docker 镜像与多服务编排
- 依赖 health check
- migration 前置
- smoke 自动化验证

### 你做 observability 时最先接哪几段链路

回答重点：

- 登录
- 发消息
- WebSocket 建连
- Redis 在线态
- 搜索索引链路

### 你怎么判断一个项目已经从“Demo”进入“工程项目”

回答重点：

- 有 schema 演进
- 有健康检查
- 有 smoke
- 有指标和 trace
- 有诊断入口
- 有明确的服务边界
