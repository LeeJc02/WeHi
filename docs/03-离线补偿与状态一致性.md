# 离线补偿与状态一致性

如果只做 WebSocket 广播，这个项目撑不起“实时系统”这几个字。因为一旦用户掉线，在线广播就丢了，消息列表、已读状态、会话预览都可能不一致。

这个仓库真正有辨识度的点，是它补了一条用户视角的增量补偿链。

## 核心设计

项目不是把离线补偿做成“重新全量拉一次”，而是做成：

- 服务端持久化 `sync_events`
- 客户端保存 `cursor`
- 重连后按 `cursor` 之后的事件增量追平

也就是：

- 在线时靠 WebSocket 低延迟推
- 离线后靠 `sync_events` 最终追平

## 服务端怎么写补偿事件

文件：

- `internal/app/chat/sync.go`
- `internal/app/repository/repository.go`

事件写入入口：

```go
func (d *dependencies) emitSyncEvent(userIDs []uint64, eventType, aggregateID string, payload any) {
	userIDs = uniqueIDs(userIDs)
	if len(userIDs) == 0 {
		return
	}
	if d.repo != nil {
		_ = d.repo.AppendSyncEvents(userIDs, eventType, aggregateID, payload)
	}
	if d.rabbit != nil {
		_ = d.publishJSON("sync.notify", contracts.SyncNotifyEvent{Recipients: userIDs})
	}
}
```

它同时做了两件事：

1. 把事件按用户写进 `sync_events`
2. 给在线客户端发一个 `sync.notify`

这意味着在线和离线路径并不冲突：

- 在线用户也可以收到通知后补拉，兜底丢事件
- 离线用户重连后仍能靠 cursor 追平

## 数据库存什么

模型定义在：

- `internal/app/repository/models.go`

```go
type SyncEvent struct {
	ID          uint64
	UserID      uint64
	EventType   string
	AggregateID string
	Payload     string
	CreatedAt   time.Time
}
```

这几个字段的意义分别是：

- `ID`：天然递增 cursor
- `UserID`：每个用户独立补偿流
- `EventType`：前端要怎么合并
- `AggregateID`：方便按消息或会话追诊断
- `Payload`：事件正文

## 查询补偿事件

文件：

- `internal/app/sync/service.go`
- `internal/app/repository/repository.go`

Repository 层：

```go
func (r *Repository) ListSyncEvents(userID, cursor uint64, limit int) ([]SyncEvent, uint64, error) {
	if err := r.db.Where("user_id = ? AND id > ?", userID, cursor).
		Order("id asc").
		Limit(limit).
		Find(&rows).Error; err != nil {
		return nil, 0, err
	}
	currentCursor, err := r.CurrentSyncCursor(userID)
	...
}
```

Service 层返回：

- `events`
- `next_cursor`
- `current_cursor`
- `has_more`

这个接口设计是标准的“增量追平分页”模式，面试时很好讲。

## 前端怎么追平

文件：

- `frontend/lib/chat-store.tsx`

关键逻辑：

```ts
const result = await getSyncEvents(cursor, 50)
for (const event of result.events) {
  applyIncomingEvent({
    type: event.event_type || event.type,
    payload: event.payload,
  })
  cursor = event.cursor
}
syncCursorRef.current = result.next_cursor || result.current_cursor || cursor
```

这说明前端不是拿到一个“全量状态快照”，而是逐条重放事件，把本地状态机推进到最新。

## 为什么这套方案适合 IM

### 相比全量刷新

优点：

- 网络成本低
- 恢复速度快
- 更适合移动端和弱网环境

### 相比只靠 MQ 重放

优点：

- 不依赖客户端直接接触消息中间件
- 用户视角的事件流更容易做权限隔离
- API 层更容易补审计、补诊断

## 一致性边界

这套方案不是“绝对强一致”，它更像实时系统常见的做法：

- 主链路尽量快
- 事件异步扩散
- 客户端重连后最终追平

所以你应该把它描述为：

- 在线场景追求低延迟
- 掉线场景依赖增量补偿保证最终一致

而不是说“任何时刻都完全强一致”。

## 面试常见追问

### 为什么不直接让前端重连后重新拉消息列表和会话列表

建议回答：

- 那样会带来大量重复查询
- 不容易恢复输入态、已读推进、会话移除等细粒度状态
- 事件流更适合表达“发生了什么变化”

### sync_events 会不会无限增长

建议回答：

- 生产环境需要做 TTL、归档或按时间窗口清理
- 当前实现重点放在补偿链路本身，清理策略还可以继续做成独立任务
- 这是明显的工程化追问点，应该主动提出来

### aggregate_id 有什么意义

建议回答：

- 用于按消息、会话、好友申请聚合同一批事件
- 便于后台诊断和旅程追踪
- 不只是一个“附加字段”，而是后续可观测性和排障的锚点
