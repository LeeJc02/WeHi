# 存储、缓存与搜索设计

这份文档讲三个主题：

1. 哪些状态进 MySQL。
2. 哪些状态进 Redis。
3. 搜索和事件总线为什么做成可切换抽象。

## MySQL：存最终事实

数据库 schema 的主干在：

- `migrations/001_initial_schema.sql`
- `internal/app/repository/models.go`

核心表包括：

- `users`
- `friend_requests`
- `friendships`
- `conversations`
- `conversation_members`
- `messages`
- `sync_events`
- `ai_audit_logs`
- `ai_retry_jobs`

为什么这些要进 MySQL？

- 用户、关系、消息、游标这些都属于需要可恢复、可审计、可查询的强事实
- 这部分不能只放 Redis，否则服务重启或 key 丢失后状态会断裂

## Conversation 和 Message 模型的设计重点

### conversations

会话表不仅存基础信息，还做了列表页聚合所需的冗余：

- `last_message_seq`
- `last_message_sender`
- `last_message_preview`
- `last_message_type`
- `last_message_at`

这说明项目在设计时考虑了“列表查询效率”，不是每次会话列表都回表扫最后一条消息。

### messages

消息模型里值得专门拿出来讲的字段：

- `seq`
- `client_msg_id`
- `status`
- `delivery_status`
- `reply_to_message_id`
- `attachment_json`
- `recalled_at`

这里的关键思想是：

- `seq` 负责会话内顺序
- `client_msg_id` 负责发送幂等
- `delivery_status` 负责客户端展示态

## Redis：存高频临时状态

### 1. 在线状态与连接路由

文件：

- `internal/app/presence/service.go`

项目里当前围绕在线用户维护高频临时状态：

```go
const onlineSetKey = "presence:online"
```

配套方法：

- `MarkOnline`
- `MarkOffline`
- `OnlineMap`
- `OnlineUsers`

这类状态适合进 Redis，因为：

- 高频读写
- 短期有效
- 可容忍重建

在系统设计上，Redis 不只承担一个 online set，还承担两类关键临时状态：

- `user -> gateway instance` 的连接路由
- resume 恢复信息和断线后的恢复上下文

### 2. 用户会话

文件：

- `internal/app/auth/service.go`

登录成功后，session 状态写入 Redis。这里不是把 refresh token 只留在客户端，而是服务端也保留会话状态，因此可以支持：

- 单端退出
- 全端退出
- 退出其他设备
- 会话列表查看

这比“只发 JWT 不存状态”更适合企业 IM。

## Kafka 事件总线

文件：

- `internal/platform/rabbit/rabbit.go`

消息系统通过事件总线解耦同步事务链和异步扩散链。核心异步事件包括：

- `message.accepted`
- `message.persisted`
- `message.delivered`
- `message.read`
- `typing.updated`
- `message.recalled`
- `sync.notify`

Kafka 在这里承接的是多类异步链路：

- 消息发送后的状态扩散
- 已读、送达、撤回等生命周期事件
- 搜索索引更新
- 离线补偿通知
- 后台诊断和审计扩展点

这套设计的价值在于：

- 业务写库和状态扩散解耦
- 消费组可以按职责拆分
- 更适合高吞吐、可回放、可扩展的事件流治理

## 搜索设计

文件：

- `internal/platform/search/elasticsearch.go`
- `internal/app/chat/search_service.go`
- `internal/app/repository/repository.go`

搜索链路采用 Elasticsearch 建索引，服务层负责根据会话权限范围收敛查询边界。

服务层明确判断：

```go
if s.deps.search.IsMock() {
	hits, err = s.deps.repo.SearchMessages(ids, query, offset, limit)
} else {
	hits, err = s.deps.search.SearchMessages(ctx, s.deps.messagesIndex, query, ids, offset, limit)
}
```

这说明搜索不是脱离业务权限独立存在的，而是和会话成员范围、消息可见范围一起工作。

## 为什么这种设计有工程价值

- 高频状态和最终事实分层清晰
- 事件总线把核心事务链和异步链解耦
- 搜索、补偿、状态同步都能接入统一事件流

## 面试可展开的点

### 为什么在线态放 Redis，不放 MySQL

建议回答：

- MySQL 更适合存事实，不适合承接高频波动
- 在线态是典型的高频临时状态
- 放 Redis 可以显著降低数据库写压力

### 为什么搜索要单独做 Elasticsearch

建议回答：

- IM 搜索对关键词、高亮、分页和扩展性要求更高
- 关系型库适合存事实，不适合承接全文检索主链路
- Elasticsearch 更适合独立承担检索能力

### 为什么要基于 Kafka 做消息总线

建议回答：

- Kafka 更适合做顺序分区、消费组扩展和高吞吐事件流
- 消息发送、送达、已读、索引、补偿都能接进统一总线
- 业务服务更专注领域逻辑而不是把异步副作用塞进同步请求
