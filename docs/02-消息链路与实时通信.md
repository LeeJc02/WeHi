# 消息链路与实时通信

这份文档聚焦最核心的问题：一条消息从发送到被别人看到，中间到底发生了什么。

## 总链路

1. 客户端调用 `POST /api/v1/conversations/:id/messages`
2. `api-service` 做成员校验、幂等校验、消息落库
3. 服务端写消息生命周期事件
4. `realtime-service` 消费事件并广播给在线连接
5. 接收方在线则实时看到消息
6. 接收方阅读后回写 `message.read`
7. 发送方收到已读回执，更新消息状态

## 为什么要拆成两条链

### 同步链

同步链负责“请求立即返回”。

- 校验
- 落库
- 返回消息 DTO

### 异步链

异步链负责“状态扩散”。

- 广播在线用户
- 更新搜索索引
- 触发离线补偿通知

这意味着接口不会被“所有下游都成功”拖死。

## 消息发送的关键代码

文件：

- `internal/app/chat/message_service.go`

消息发送的入口会先做成员校验、消息类型校验和幂等控制：

```go
if _, err := s.deps.repo.FindConversationMember(conversationID, userID); err != nil {
	return nil, apperr.Forbidden("CONVERSATION_MEMBERSHIP_REQUIRED", "not a conversation member")
}

if messageType != "text" && messageType != "system" && messageType != "image" && messageType != "file" {
	return nil, apperr.BadRequest("UNSUPPORTED_MESSAGE_TYPE", "unsupported message_type")
}

if clientMsgID == "" {
	clientMsgID = fmt.Sprintf("%d-%d", conversationID, time.Now().UnixNano())
}
```

这里有两个值得讲的点。

### 1. client_msg_id 幂等

它不是简单生成 message id，而是为“重试不重复发”服务。

数据库层也补了唯一约束调整：

- `migrations/006_message_lifecycle_idempotency.sql`

```sql
ALTER TABLE messages
  ADD UNIQUE KEY uk_messages_sender_client_msg_id (sender_id, client_msg_id);
```

这意味着：

- 同一个发送方重复提交相同 `client_msg_id` 不会插出多条消息
- 可以支撑弱网重试、前端重复提交、移动端重放

### 2. Reply / attachment 统一进入消息模型

不是另做一套引用消息系统，而是在消息模型内挂：

- `reply_to_message_id`
- `attachment_json`

这样列表查询时就可以统一补 `reply_to` 和附件 DTO。

## 实时投递

文件：

- `services/realtime/main.go`
- `internal/realtime/hub.go`

`realtime-service` 会消费多类事件：

- `message.accepted`
- `message.persisted`
- `message.delivered`
- `message.read`
- `typing.updated`
- `message.recalled`
- `sync.notify`

广播核心代码在 `internal/realtime/hub.go`：

```go
for _, userID := range userIDs {
	for conn := range h.conns[userID] {
		_ = conn.SetWriteDeadline(time.Now().Add(10 * time.Second))
		if err := conn.WriteJSON(event); err != nil {
			stale = append(stale, ...)
		}
	}
}
```

这段代码体现了：

- 一个用户允许多连接
- 广播按 `userID -> connections` 组织
- 写失败连接会被惰性清理

## WebSocket 建连与保活

文件：

- `internal/controllers/realtime_controller.go`

关键点：

- access token 通过 `/ws?token=...` 解析
- 连接成功后立即回 `auth.ok`
- 服务端周期性发 ping
- 客户端 pong 用于续期 read deadline
- 断开时会移除连接并清理在线态

核心代码：

```go
const (
	wsWriteWait = 10 * time.Second
	wsPongWait  = 60 * time.Second
	wsPingEvery = (wsPongWait * 9) / 10
)
```

这说明它不是裸连，而是明确做了保活窗口设计。

## 消息送达与已读

### delivered

项目里 `delivered` 不是数据库轮询推出来的，而是结合在线态判断。

文件：

- `internal/app/chat/message_service.go`
- `internal/app/presence/service.go`

逻辑是：

1. 发送后查询会话其他成员
2. 到 Redis 查这些成员是否在线
3. 只要存在在线接收方，就把消息状态推进到 `delivered`

这不是最终一致的“强送达证明”，但非常适合 IM 前台语义。

### read

已读推进不是单条消息逐条写，而是维护 `conversation_members.last_read_seq`：

```go
return r.db.Model(&ConversationMember{}).
	Where("conversation_id = ? AND user_id = ? AND deleted_at IS NULL", conversationID, userID).
	Update("last_read_seq", gorm.Expr("GREATEST(last_read_seq, ?)", seq)).Error
```

这个设计很重要，因为它有两个优势：

- 不需要逐条更新消息表
- 天然支持“读到某条为止”的批量已读语义

## 前端如何合并事件

文件：

- `frontend/lib/chat-store.tsx`

前端不是简单 `onmessage => append`，而是按事件类型精细合并：

- `message.accepted`：把本地 sending 改成 sent
- `message.persisted`：插入新消息并更新会话列表
- `message.delivered`：推进发送方消息状态
- `message.read`：推进 last_read_seq 和发送方消息读状态
- `typing.updated`：维护输入态用户列表
- `message.recalled`：把消息和会话预览改成已撤回

这部分很适合面试里讲“前后端状态同步是怎么设计的”。

## 面试最容易被追问的点

### 为什么 WebSocket 单独拆服务

建议回答：

- 因为长连接和普通 REST 请求的资源模型不同
- 网关服务更关注连接生命周期和广播效率
- 业务服务更关注校验、落库和事务
- 拆开以后更容易水平扩容

### 为什么已读不用更新每条消息

建议回答：

- 每条消息更新会带来高写放大
- 会话级 `last_read_seq` 足够表达“读到哪里”
- 反推未读数和消息状态都更高效

### client_msg_id 为什么放在发送方维度唯一

建议回答：

- 全局唯一成本更高
- `sender_id + client_msg_id` 已足够表达同一个客户端的一次发送行为
- 更适合弱网重试和端上重放
