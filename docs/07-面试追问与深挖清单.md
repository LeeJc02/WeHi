# 面试追问与深挖清单

这份文档专门面向“拷打式追问”。问题分成两类：

- 工程追问：围绕架构、性能、故障、扩展性
- 过程追问：围绕设计选择、权衡、推进方式、踩坑

每个问题后面都给出回答方向，不追求背稿，重点是知道该往哪里讲。

## 一、架构追问

### 1. 为什么要拆成三个服务，而不是一个单体

回答方向：

- 认证、业务 API、实时网关的资源模型完全不同
- WebSocket 长连接和普通 REST 请求放一起，连接管理会污染业务服务
- 拆分后更容易单独扩容 realtime-service
- 主函数只做依赖装配，业务边界更清晰

证据文件：

- `services/auth/main.go`
- `services/api/main.go`
- `services/realtime/main.go`

### 2. 为什么实时网关不直接读数据库，而要消费事件

回答方向：

- 广播服务不应该承担业务事务
- 事件驱动可以把“写库”和“状态扩散”解耦
- 可以自然挂更多异步消费者，例如搜索索引、审计、补偿

### 3. 为什么已读用 `last_read_seq`，不用每条消息单独标记

回答方向：

- 每条消息单独更新写放大会很大
- 会话级游标足够表达“读到哪里”
- 更适合推导未读数和发送方消息状态

对应文件：

- `internal/app/repository/repository.go`

### 4. 为什么需要 `sync_events`

回答方向：

- WebSocket 只能覆盖在线用户
- 断线期间的状态变更不能靠广播补回来
- `sync_events + cursor` 可以把离线恢复变成增量追平

### 5. 为什么 `sync_events` 按用户存，而不是按会话存

回答方向：

- 客户端恢复时本质上是“恢复这个用户的视图”
- 用户维度更适合做权限隔离
- 会话维度虽然更省存储，但恢复时还得再做一次用户过滤

### 6. 为什么在线态、路由态和 resume 信息要放 Redis

回答方向：

- 这类状态高频变动、生命周期短，不适合放 MySQL
- 多实例网关需要快速定位用户连接所在实例
- resume 信息适合放在低延迟缓存里，便于断线恢复

这道题适合顺势讲“为什么 Redis 是实时系统里的状态中枢”。

## 二、事件总线追问

### 7. 为什么消息系统要基于 Kafka 做事件总线

回答方向：

- Kafka 更适合顺序分区、消费组扩展、堆积削峰
- 消息发送、送达、已读、索引、补偿都可以放在统一事件流里
- 可以把同步事务链和异步副作用链清晰拆开

如果面试官继续追：

- 分区键可以按会话或业务主键设计
- 不同消费组分别承担广播、索引、审计、补偿等职责

### 8. 如果 MQ 挂了怎么办

回答方向：

- 主链路首先保证消息先落库
- 实时通知和搜索索引属于异步副作用
- Kafka 故障会影响状态扩散，但不会直接丢失消息事实
- 可以依赖补偿机制、重试和索引重建恢复

### 9. 事件重复消费怎么办

回答方向：

- 发送侧有 `client_msg_id` 幂等
- 消费侧需要业务幂等，尤其是索引和补偿事件
- 系统的消息主表和事件链路都需要体现幂等意识

## 三、数据库与性能追问

### 10. 会话列表为什么能比较快

回答方向：

- 会话表冗余了最后一条消息信息
- unread_count 通过聚合查询算出
- pinned / muted / draft 通过 settings 表按用户维度挂载

### 11. 大群聊下未读数统计会不会慢

回答方向：

- 会，尤其是完全依赖实时 SQL 聚合时
- 当前设计更适合中等规模场景
- 如果规模继续扩大，需要把未读数、会话统计进一步做缓存化或异步物化

### 12. 为什么搜索要独立使用 Elasticsearch

回答方向：

- 消息搜索对全文检索、分页和扩展性要求更高
- 关系型库适合存事实，不适合承担主搜索链路
- Elasticsearch 更适合作为独立检索层

## 四、长连接追问

### 13. WebSocket 断开后如何恢复

回答方向：

- 前端重建连接
- 连接恢复后先触发 `sync.notify` 或主动走 `getSyncCursor/getSyncEvents`
- 再用事件流追平本地状态

### 14. 心跳机制是怎么做的

回答方向：

- 服务端定时发 ping
- 客户端 pong 续 read deadline
- 超时或写失败就清理连接

对应文件：

- `internal/controllers/realtime_controller.go`

### 15. 一个人多端在线怎么处理

回答方向：

- Hub 结构是 `userID -> connections`
- 同一用户允许多个连接
- 广播按用户发，天然支持多端同步

## 五、工程化追问

### 16. 你怎么保证多服务项目既能本地联调，又能线上稳定部署

回答方向：

- 本地通过统一脚本和容器链路快速启动
- 线上通过 Docker + Kubernetes 统一交付
- migration 前置
- health check
- 统一脚本
- smoke

### 17. 为什么不用 AutoMigrate

回答方向：

- 线上 schema 变更要可审计
- 索引和约束的调整不能依赖黑盒自动迁移
- migration 文件更利于 review 和回滚

### 18. 你做了哪些观测能力，不是只说了 OTel 这个词

回答方向：

- trace：HTTP、DB、Redis、WebSocket
- metric：请求量、延迟、WS 连接数
- log：结构化访问日志
- 诊断：消息旅程、会话一致性、AI 审计

## 六、过程追问

### 19. 你是怎么决定先做哪一层的

回答方向：

- 先打基础骨架：服务拆分、配置、路由、migration
- 再做主链路：登录、会话、消息
- 再补实时链路：WebSocket、事件广播
- 最后补工程能力：搜索、补偿、观测、后台诊断

这样回答会显得你有推进顺序感。

### 20. 这个项目里你觉得最难的点是什么

推荐回答：

- 不是单个接口实现
- 而是如何把在线广播和离线补偿同时讲通
- 也就是“低延迟”和“最终一致”两条链怎么配合

### 21. 你踩过什么坑

可选回答方向：

- 如果没有 `client_msg_id`，弱网重试容易打出重复消息
- 如果没有 `sync_events`，掉线重连只能全量刷新，状态恢复很差
- 如果把在线态直接打数据库，压力和抖动都会很明显
- 如果服务边界不清，WebSocket 和业务 API 会越写越耦合

### 22. 你最后还想继续做什么

可以讲这些：

- 给 `sync_events` 补清理与归档
- 优化 Kafka topic 治理和消费组拆分
- 继续细化在线态、多实例路由和 resume 恢复策略
- 给搜索补更真实的索引治理和恢复策略

## 七、面试时最危险的说法

下面这些说法尽量不要直接说。

- “这个系统已经完全强一致。”
- “消息送达是严格可靠投递。”
- “我这个搜索就跟生产一样。”

更稳的说法应该是：

- “一致性边界在哪里”
- “如果继续扩展，我会怎么做”

## 八、最后的回答原则

面试官越强，越不怕你承认边界，越怕你混淆边界。

所以这类项目最好的回答方式是：

1. 先讲整体架构。
2. 再讲一条主链路。
3. 再讲一条你最有辨识度的设计。
4. 最后主动讲边界、风险和优化空间。

这样比堆技术名词更有说服力。
