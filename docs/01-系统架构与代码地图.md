# 系统架构与代码地图

这份文档回答三个问题：

1. 项目到底拆成了哪些服务。
2. 每层代码分别负责什么。
3. 面试时说“我设计了后端架构”时，证据落在哪些文件。

## 服务拆分

项目入口明确拆成三个服务：

- `services/auth/main.go`
- `services/api/main.go`
- `services/realtime/main.go`

职责边界如下。

### auth-service

- 用户注册
- 登录与 refresh token 轮换
- 设备会话管理
- JWT 鉴权中间件

核心文件：

- `internal/app/auth/service.go`
- `internal/routes/auth.go`
- `internal/controllers/auth_controller.go`

### api-service

- 用户、好友、会话、消息、搜索
- 后台管理、AI、诊断、补偿入口
- 写入消息事件与补偿事件

核心文件：

- `internal/routes/api.go`
- `internal/app/chat/*.go`
- `internal/app/sync/service.go`
- `internal/controllers/*.go`

### realtime-service

- WebSocket 建连
- 心跳、保活、在线态写入
- 消费事件并广播给在线用户

核心文件：

- `internal/controllers/realtime_controller.go`
- `internal/realtime/hub.go`
- `services/realtime/main.go`

## 分层设计

### Controller 层

职责是参数绑定、鉴权入口、HTTP 响应格式统一。

目录：

- `internal/controllers`

比如：

- `message_controller.go` 负责消息发送、已读、输入态、撤回
- `conversation_controller.go` 负责建会话、改名、加人、退群

### Service 层

职责是业务规则，不直接暴露 HTTP 细节。

目录：

- `internal/app/auth`
- `internal/app/chat`
- `internal/app/sync`
- `internal/app/admin`
- `internal/app/presence`

比如消息发送的业务规则就在 `internal/app/chat/message_service.go`：

```go
if _, err := s.deps.repo.FindConversationMember(conversationID, userID); err != nil {
	return nil, apperr.Forbidden("CONVERSATION_MEMBERSHIP_REQUIRED", "not a conversation member")
}
if content == "" && attachment == nil {
	return nil, apperr.BadRequest("MESSAGE_CONTENT_REQUIRED", "content is required")
}
```

这段代码体现的是：

- 不是 Controller 在决定权限
- 不是 Repository 在决定业务规则
- 业务边界被压在 Service 层

### Repository 层

职责是聚合查询、事务、持久化。

目录：

- `internal/app/repository`

比如会话列表不是多次查询拼出来，而是一条聚合 SQL 拉出：

```sql
SELECT
	c.id,
	c.type,
	c.name,
	member_stats.member_count,
	COALESCE(unread_stats.unread_count, 0) AS unread_count
FROM conversation_members cm
JOIN conversations c ON c.id = cm.conversation_id
...
ORDER BY
	CASE WHEN cs.pinned_at IS NULL THEN 1 ELSE 0 END,
	cs.pinned_at DESC,
	c.updated_at DESC
```

对应文件：

- `internal/app/repository/repository.go`

### Platform 层

职责是数据库、Redis、消息总线、可观测性、中间件等底层能力。

目录：

- `internal/platform/db`
- `internal/platform/redis`
- `internal/platform/rabbit`
- `internal/platform/search`
- `internal/platform/observability`
- `internal/platform/httpx`

## 代码地图

### 配置装配入口

- `internal/config/config.go`

这里统一收口：

- MySQL DSN
- Redis
- Kafka / MQ 配置
- Elasticsearch URL
- JWT
- OTel

### 路由装配

- `internal/routes/auth.go`
- `internal/routes/api.go`
- `internal/routes/realtime.go`

建议面试时这样描述：

“我把服务入口、路由装配、业务服务、数据访问和平台基础设施分开了，主函数只做依赖注入，不承载业务逻辑。”

### DTO 与事件协议

- `pkg/contracts/requests.go`
- `pkg/contracts/responses.go`
- `pkg/contracts/queries.go`
- `pkg/contracts/events.go`

这部分很重要，因为它体现了项目不是“Controller 里到处写匿名结构体”，而是把协议层收拢成了独立契约。

## 架构设计的重点表达

这套架构的核心不是把功能拆散，而是把不同运行特征的职责拆开：

- `auth-service` 负责鉴权与会话治理
- `api-service` 负责同步请求、事务和领域规则
- `realtime-service` 负责长连接、在线状态、事件消费与广播

再往下看，平台层统一收口 Redis、Kafka、搜索和可观测性能力，业务层只处理领域逻辑。这种拆法很适合在简历和面试里表达“后端架构设计与核心链路实现”。
